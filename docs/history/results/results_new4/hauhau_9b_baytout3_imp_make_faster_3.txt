GOAL
Optimize the application's initial page load time (Time to Interactive) by reducing client-side bundle size, minimizing critical rendering path blocking resources, and implementing efficient data fetching strategies. Target a reduction in First Contentful Paint (FCP) from current baseline to under 1.5s on average network conditions.

FILES
- `src/main.tsx` / `index.html`: Entry points for bundling optimization and meta tags.
- `package.json`: Dependencies audit for tree-shaking opportunities or lighter alternatives.
- `vite.config.ts` (or equivalent build config): Configuration for code splitting, lazy loading, compression settings.
- `src/components/Dashboard/`.tsx: Components requiring memoization or virtual scrolling if data-heavy.
- `public/assets/`: Static assets like hero images, icons, and fonts to be optimized/compressed.

STEPS
1. **Bundle Analysis**: Run `npm run build` with the specific analysis plugin (e.g., `rollup-plugin-visualizer`) to identify top 5 largest modules by size.
2. **Code Splitting & Lazy Loading**: Refactor heavy dashboard widgets into separate chunks and implement dynamic imports (`import()`) for non-critical components, ensuring they load only when viewed or scrolled into view.
3. **Critical CSS Extraction**: Extract above-the-fold styles from the main stylesheet to an inline `<style>` block in `index.html` before loading the full CSS file. Defer non-critical CSS using media queries (`media="print"` with onload event).
4. **Asset Optimization**: Convert all images used on the dashboard landing view to WebP/AVIF format and implement responsive image sources via `<picture>`. Enable Brotli compression in the build config for text assets (JS/CSS) if not already active.
5. **HTTP Caching Strategy**: Update `vite.config.ts` or server middleware to set appropriate cache headers (`Cache-Control: max-age=31536000, immutable`) for hashed static asset files and long-lived fonts/icons.
6. **Data Fetching Optimization**: Replace synchronous API calls with asynchronous patterns using React Query (TanStack) or SWR, implementing a "skeleton UI" state before data returns to prevent layout shift (CLS). Add request cancellation logic when the user navigates away during loading.

EDGE CASES
- **Slow 3G Networks**: Ensure fallback images and minimal critical CSS are served first; verify lazy-loaded chunks do not block initial interaction on high-latency connections.
- **Browser Compatibility**: Test critical rendering path optimizations (e.g., `loading="lazy"`, `<link rel="preload">`) across Safari, Firefox, and Edge to ensure no regression in older versions lacking specific native support.
- **Large Datasets**: If the dashboard renders a data table with >100 rows initially, implement virtual scrolling or pagination immediately after load rather than rendering all DOM nodes at once.

ACCEPTANCE
- Lighthouse Performance score for "Initial Load" increases by minimum 20 points (or FCP drops below 1.5s).
- Bundle size reduced by at least 30% via code splitting and tree-shaking of unused dependencies in the initial chunk.
- No Layout Shift (CLS) > 0.1 observed during the transition from loading state to data population.
- All lazy-loaded components render within 200ms after their respective network requests complete on a standard broadband connection.
