GOAL
Isolate and resolve the authentication failure in the application by identifying whether the root cause lies in token generation, validation logic, database schema mismatches, or network configuration between services (e.g., Auth0, Okta, AWS Cognito). The fix must ensure that valid credentials result in a successful session establishment without introducing race conditions or breaking existing user flows.

FILES
- `src/auth/` directory contents: specifically `token.service.ts`, `user.controller.ts`, and any middleware handling JWT verification (e.g., `auth.middleware.js`).
- Database migration scripts related to the users table schema (`schema.sql`, `migrations/user_20231027_add_token_expiry.sql`).
- Environment configuration files containing secrets or endpoint URLs: `.env.production`, `config/auth.config.ts`.
- Unit test suites for authentication logic: `tests/unit/auth.spec.js` and integration tests in `e2e/login.cy.ts`.

STEPS
1. Reproduce the failure by executing a login request with known valid credentials using Postman or curl, capturing the exact HTTP status code (401/500) and response body payload.
2. Inspect server-side logs (`logs/app.log`) immediately preceding the error to identify stack traces, database query failures, or timeout exceptions.
3. Verify JWT signing keys in `.env` match those expected by the client application; ensure `NODE_ENV=production` is set correctly if using different keys for dev/prod.
4. Check database connectivity and schema: confirm that the `users` table has columns matching the token payload claims (e.g., `email`, `password_hash`) and that indexes exist on authentication fields to prevent query timeouts under load.
5. Review middleware logic in `auth.middleware.js`: ensure headers are parsed correctly, expiration dates (`exp`) are validated against current UTC time, and refresh tokens are handled if applicable.
6. If using an external provider (e.g., Auth0), verify the callback URL matches exactly what is registered in the dashboard to prevent redirect loops or silent failures.
7. Implement a fix based on findings: update environment variables, patch SQL queries, adjust middleware logic, or regenerate signing keys.
8. Run existing unit tests (`npm test`) and integration tests (`npx cypress open`) to confirm no regressions were introduced.

EDGE CASES
- Time skew between client machine/server causing premature token expiration errors (handle with `Date.now()` vs server time).
- Concurrent login attempts from the same user resulting in race conditions during session creation or password updates.
- Special characters in email addresses or passwords that are improperly URL-encoded before sending to an external provider.
- Database connection pool exhaustion leading to intermittent 503 errors mistaken for auth failures.
- Case sensitivity issues when comparing usernames/emails against database records (e.g., `User` vs `user`).

ACCEPTANCE
- A user can successfully log in with valid credentials across all supported browsers and devices without console errors or network timeouts.
- Server logs show no new exceptions related to authentication during a 24-hour load test simulating peak traffic.
- Unit tests for token generation, validation, and middleware pass with 100% coverage on auth-related functions.
- The application handles invalid/expired tokens gracefully by returning appropriate HTTP 401 responses instead of crashing or redirecting incorrectly.
