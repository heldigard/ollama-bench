GOAL: Optimize the frontend and backend performance of the main analytics dashboard to reduce Time to Interactive (TTI) and Largest Contentful Paint (LCP). The target is a 50% reduction in initial load time.

FILES:
- /frontend/src/components/Dashboard.tsx (Main layout)
- /frontend/src/hooks/useDataFetcher.ts (API interaction logic)
- /backend/routes/analytics.js (Database query endpoints)
- /frontend/public/assets/ (Static assets)

STEPS:
1. Audit Performance: Run a Lighthouse audit and Chrome DevTools Network tab analysis to identify bottlenecks (e.g., large payloads, render-blocking JS, or slow DB queries).
2. Implement Pagination/Infinite Scroll: Modify the dashboard to fetch data in chunks of 50 records instead of loading the entire dataset at once.
3. Data Aggregation Optimization: Rewrite backend SQL queries to perform aggregations (SUM, AVG) at the database level rather than processing raw arrays in the Node.js environment.
4. Frontend Caching: Integrate React Query or SWR to implement stale-while-revalidate caching for dashboard widgets.
5. Image & Asset Optimization: Convert heavy hero images to WebP format and implement lazy loading for non-critical components below the fold.
6. Code Splitting: Implement dynamic imports (React.lazy) for secondary dashboard widgets that are not visible on initial load.

EDGE CASES:
- Large Data Sets: Ensure the UI remains responsive when a user requests a date range spanning multiple years.
- Slow Network Connections: Provide skeleton loaders or "loading" states to prevent layout shifts during data fetching.
- Concurrent Requests: Handle race conditions where multiple widget fetches might return out of order.

ACCEPTANCE:
- LCP (Largest Contentful Paint) is under 2.5 seconds on a standard 4G connection.
- API response time for the primary "Overview" widget is < 300ms.
- No layout shifts (CLS < 0.1) occur during data hydration.
- Dashboard remains functional when offline or in low-bandwidth environments via cached data.
