GOAL
Isolate and resolve authentication failure in the user login flow where valid credentials result in a 401 Unauthorized or silent redirect to the dashboard without session creation. The fix must ensure that after successful credential verification, an access token is generated (if using JWT) or a server-side session cookie is set with correct `HttpOnly`, `Secure` (in production), and `SameSite=Lax/Strict` attributes. If using OAuth2/OIDC, the callback handler must correctly exchange the authorization code for tokens without redirecting prematurely on error states.

FILES
1. `/src/routes/auth/login.ts` (or equivalent controller file) - Primary entry point for POST /login or GET /auth/callback.
2. `/src/services/user.service.ts` - Contains logic to validate credentials against database and generate tokens/sessions.
3. `/src/middleware/session.middleware.ts` (if applicable) - Handles cookie serialization/deserialization.
4. `.env` - Check for `JWT_SECRET`, `SESSION_SECRET`, or OAuth client secrets that might be mismatched in staging vs production causing silent failures.

STEPS
1. Add logging to the login endpoint immediately before and after credential validation: log raw request body, user ID found (masked), timestamp, and error stack trace if thrown.
2. Verify database query execution by inserting a `debug` flag into the SQL/NoSQL condition; confirm the correct row is retrieved for known test users.
3. Inspect token generation logic: ensure the payload includes standard claims (`sub`, `iat`, `exp`) and that the signing algorithm matches what the client expects (e.g., HS256 vs RS256). Re-sign tokens with a fresh secret to rule out stale key rotation issues.
4. Check response headers on successful login using browser DevTools Network tab: verify `Set-Cookie` presence, domain/path attributes, and Content-Type (`application/json`). If returning JSON but not setting cookies, ensure the framework (Express/NestJS/Fastify) is configured to send them explicitly or via a helper like `res.cookie()`.
5. Test with Postman/curl: Send request headers including any required CSRF tokens or Origin checks; verify response status code changes from 401/302 to 200 and inspect the body for token payload structure.
6. If using an external IDP (Auth0/Cognito/AWS Cognito), check console logs in the provider dashboard for "Silent Logout" events or mismatched redirect URIs causing callback failures.

EDGE CASES
- User account locked due to failed login attempts: Ensure rate limiting middleware isn't blocking valid users after 3 consecutive fails without resetting the counter on successful re-auth.
- Time skew between server and client: If using JWT with `exp` claims, ensure NTP sync is active; a 5-minute drift can cause immediate expiration before use.
- Case sensitivity in username/email fields: Verify database collation matches frontend input (e.g., "Admin" vs "admin").
- Special characters in passwords/emails causing SQL injection or regex escape issues during validation.
- Browser cookie policies blocking `Secure` cookies when testing over HTTP locally; temporarily disable the strict flag for local dev to confirm logic works, then re-enable with HTTPS tunneling (ngrok/localtunnel).

ACCEPTANCE
1. POST /login returns 200 OK and includes a valid JWT in the response body or sets an HttpOnly cookie within 5 seconds of request receipt.
2. Decoding the returned token on the client side yields `sub` matching the logged-in user ID, with `exp > current_time + buffer`.
3. Refreshing the page without re-entering credentials maintains session state (if using cookies) or allows silent refresh via `/refresh` endpoint.
