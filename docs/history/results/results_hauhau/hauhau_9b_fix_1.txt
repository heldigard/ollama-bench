GOAL
Isolate and resolve a critical authentication failure preventing users from logging in or maintaining sessions after an update. The fix must ensure secure token validation, correct session management, proper error handling for invalid credentials/tokens, and seamless integration with the existing user database without introducing race conditions or security regressions (e.g., timing attacks).

FILES
1. `src/auth/login.ts` (or equivalent login handler file) - Primary focus for credential verification logic.
2. `src/middleware/auth.middleware.ts` - Focus for session/token validation middleware and header parsing.
3. `config/database.config.json` or environment variables related to JWT secret/session store connection strings.
4. `tests/unit/auth.spec.ts` - To verify the fix against existing test cases.

STEPS
1. **Reproduce**: Trigger a login request with valid credentials using Postman/Insomnia and observe the exact error response (e.g., 500 Internal Server Error, 401 Unauthorized) or behavior (infinite redirect loop). Check server logs for stack traces related to `jwt.decode`, `bcrypt.compare`, or database connection timeouts.
2. **Analyze Logs**: Correlate request timestamps with log entries to identify if the failure occurs during password hashing, token generation, session storage, or response serialization. Look for specific error codes (e.g., "Token expired", "User not found", "Session store full").
3. **Inspect Code Flow**: Trace the execution path from the API endpoint through middleware layers. Verify that asynchronous operations (DB queries) are properly awaited and that exceptions in one layer do not swallow errors in another. Check for hardcoded secrets or incorrect environment variable usage if applicable.
4. **Implement Fix**: Apply code changes based on findings:
   - If token validation fails, ensure `verify` options include correct clock skew tolerance and algorithm checks.
   - If DB query fails, add proper error handling to distinguish between "user not found" vs "database down".
   - If session storage is the issue, increase capacity or switch from memory-based store (Redis/Memcached) if scaling issues are detected.
5. **Update Middleware**: Ensure `req.user` and `res.locals.token` are correctly populated before passing to protected routes. Add fallback logic for missing headers.
6. **Test Locally**: Run the application with a fresh database state, perform login/logout cycles, test edge cases (empty password, special characters in username), and verify JWT expiration handling.

EDGE CASES
1. **Concurrent Logins**: Ensure session tokens are unique per user/device to prevent one token from hijacking another's active session if using shared secrets improperly.
2. **Database Latency/Timeouts**: Handle cases where the DB query takes longer than expected (e.g., >50ms) without timing out the HTTP request prematurely, potentially causing a 401 before password verification completes.
3. **Time Skew**: Account for server clock drift when validating `exp` and `nbf` claims in JWTs; ensure tolerance is consistent across all services.
4. **Special Characters/Encoding**: Test usernames/passwords with Unicode characters, spaces, or SQL injection payloads to prevent encoding errors during DB insertion/validation.
5. **Rate Limiting**: Verify that repeated failed login attempts trigger rate limiting correctly and don't lock out users due to a bug in the counter reset
