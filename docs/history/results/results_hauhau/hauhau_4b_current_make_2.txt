GOAL
Optimize frontend performance to reduce initial page load time (FCP/LCP) and improve Time to Interactive (TTI) for the main application dashboard by implementing code splitting, lazy loading non-critical assets, optimizing render paths, and enabling aggressive caching strategies.

FILES
- `src/App.tsx` / `index.tsx`: Root entry point; needs dynamic import configuration.
- `src/components/DashboardHeader.tsx`, `Sidebar.tsx`: Static components to be hoisted or inlined if small enough (analyze bundle size).
- `src/pages/ReportsPage.tsx`, `SettingsPage.tsx`: Heavy modules requiring lazy loading via React.lazy and Suspense.
- `public/static/images/*.png, *.svg`: Image assets; requires WebP conversion and responsive sizing configuration.
- `webpack.config.js` or `vite.config.ts`: Build tool config for code splitting thresholds and caching rules.

STEPS
1. **Bundle Analysis & Splitting**: Run a bundle analysis (e.g., via webpack-bundle-analyzer) to identify the largest chunks. Configure dynamic imports (`import()` with `then`) in `DashboardHeader` and heavy pages (`ReportsPage`, `SettingsPage`) to load them only after navigation or interaction, excluding core shell code from initial render.
2. **Code Splitting Thresholds**: Adjust split chunk size limits (default 10-50KB) based on analysis results; increase limit for critical CSS/JS that must be inlined immediately.
3. **Asset Optimization**: 
   - Convert all dashboard images to WebP format with fallback PNG/SVG where necessary.
   - Implement `loading="lazy"` attribute or Intersection Observer pattern for off-screen image components within the dashboard layout.
4. **Caching Strategy Update**: Configure service worker (`src/service-worker.js` or Vite plugin) to cache static assets (JS/CSS/images) with a long TTL and implement "Stale-While-Revalidate" logic where applicable. Ensure `Cache-Control: no-cache, max-age=0` is set for user-specific data endpoints while keeping UI shells cached indefinitely if hash-based.
5. **Render Optimization**: 
   - Remove unnecessary re-renders in the dashboard shell by wrapping static sections with `React.memo`.
   - Implement virtualization (windowing) for any list components within the dashboard that render >100 items to prevent DOM node explosion on initial load.

EDGE CASES
- **Network Fluctuations**: Ensure lazy-loaded chunks do not cause "Flash of Unstyled Content" or layout shifts if they fail to download; implement skeleton loaders with fallback states immediately upon component mount before data arrives.
- **Service Worker Offline Mode**: Verify that the service worker gracefully handles offline scenarios by serving cached dashboard shell and critical assets, while queuing non-critical updates for retry on next connection.
- **Large Data Sets**: If a specific report view loads 10k+ rows initially via API before pagination is implemented, ensure virtualization or infinite scroll logic triggers immediately to prevent browser freeze during the first render cycle.

ACCEPTANCE
- Lighthouse Performance score increases from baseline (e.g., <50) to >90 on Chrome DevTools.
- First Contentful Paint (FCP) reduces by at least 40% compared to current implementation.
- Time to Interactive (TTI)
