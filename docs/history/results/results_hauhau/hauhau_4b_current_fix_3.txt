GOAL: Resolve authentication failures in the user login flow by ensuring session tokens are correctly validated and persisted across API requests without requiring re-authentication for valid users.

FILES: `src/auth/middleware.ts`, `src/routes/login.route.ts`, `src/services/session.service.ts`

STEPS:
1. Audit `session.service.ts`: Verify that token expiration logic uses the correct timestamp format (ISO 8601 vs Unix epoch) and ensure database storage includes a unique session ID column if missing.
2. Update `middleware.ts`: Implement strict header validation for Bearer tokens, adding explicit checks for empty strings or malformed headers before attempting decryption.
3. Refactor `login.route.ts`: Ensure the response payload explicitly returns a new JWT token upon successful authentication and update client-side logic to store this token in HTTP-only cookies rather than localStorage (to mitigate XSS).
4. Add logging: Insert structured logs at every auth failure point, capturing user ID, IP address, error code, and specific exception message for debugging rate-limiting vs credential issues.

EDGE CASES:
- Token expiration during active session handling (requires automatic refresh mechanism or immediate re-auth prompt).
- Concurrent login attempts from different devices causing token conflicts.
- Network timeouts resulting in partial JWT payload reception.
- Invalid character encoding in the Authorization header due to browser proxy settings.

ACCEPTANCE:
- Users can log in successfully with valid credentials and receive a session cookie immediately.
- Existing authenticated users remain logged in across subsequent API calls without re-entering credentials (verified via `GET /api/user/profile`).
- Failed login attempts do not result in 401 errors for previously expired tokens if the refresh flow is triggered correctly; instead, they return appropriate error codes with clear messages.
