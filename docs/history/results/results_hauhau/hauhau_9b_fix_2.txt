GOAL
Isolate and resolve a critical authentication failure preventing user login or session persistence. The fix must ensure that valid credentials result in an authenticated state with proper token issuance (JWT/OAuth2) while invalid attempts trigger correct error responses without leaking internal stack traces. If the root cause is unknown, implement defensive logging to capture request payloads, headers, and response codes for analysis before deploying a hotfix or patch.

FILES
1. `src/auth/login.service.ts` (or equivalent controller/service handling credential validation).
2. `src/config/security.config.ts` (environment variables for secret keys, token expiration, salt rounds).
3. `.env.example` (to ensure secrets are rotated if the bug involves hardcoded defaults).
4. `Dockerfile` or deployment config (if the issue is environment-specific like missing SSL certs or wrong port binding causing timeouts during auth handshake).

STEPS
1. **Reproduce**: Spin up a fresh instance with current code and attempt login using known valid/invalid credentials to confirm failure mode (e.g., 500 Internal Server Error, infinite redirect loop, or silent drop of request).
2. **Log Analysis**: Inject debug logging into the auth service to print:
   - Incoming `Authorization` header content type.
   - Decoded payload from body/params.
   - Database query results for user lookup (check for SQL injection sanitization or N+1 issues).
   - Token generation logic output (e.g., expired secret key, mismatched algorithm like HS256 vs RS256).
3. **Root Cause Verification**: Check common failure points:
   - Secret Key rotation causing signature verification to fail on existing tokens.
   - Time skew between server and client leading to `exp` claim expiration immediately after login.
   - Race condition in session store (Redis/MongoDB) where concurrent requests overwrite valid sessions with invalid ones.
4. **Implementation**: Apply the specific fix based on findings:
   - If signature mismatch: Update `.env` secret or regenerate JWTs for existing users via a migration script.
   - If time skew: Add `clockSkewSeconds` buffer to token validation logic in `login.service.ts`.
   - If SQL injection/lookup failure: Wrap DB calls in try-catch blocks and sanitize inputs before hashing comparison.
5. **Validation**: Run automated tests against the auth endpoint with edge cases (empty body, special characters, case-insensitive username) to ensure stability.

EDGE CASES
- User account locked due to failed attempts but session token still valid from a previous successful login.
- Browser sending credentials over HTTP instead of HTTPS causing man-in-the-middle interception before server validation.
- Database connection pool exhaustion during high-concurrency auth spikes leading to timeout errors masked as "user not found".
- Multi-factor authentication (MFA) flow interruption where the initial SSO token expires before MFA verification completes.

ACCEPTANCE
1. **Happy Path**: Valid username/password returns `200 OK` with a valid access/refresh token pair within 50ms latency.
2. **Invalid Credentials**: Returns `401 Unauthorized` or `403 Forbidden` with JSON body `{ "error": "invalid_credentials" }`, no stack trace exposed in response headers/body.
3. **Rate Limiting**: After N failed attempts (configurable, default
